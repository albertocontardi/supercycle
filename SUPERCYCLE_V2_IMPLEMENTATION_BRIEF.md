## 1. PREDICTIVE SHIELD — EVOLUZIONE A 6 LIVELLI

Il file `skills/predictive-shield/SKILL.md` va riscritto per supportare 6 livelli di predizione. I livelli si attivano in cascata — ogni livello aggiunge informazione al report della shield. Il report finale è unico, non 6 report separati.

### Livello 1 — Pattern Matching Storico (già esistente, da mantenere)

Confronta i task signal del task corrente contro il Pattern Index. Funziona per keyword overlap. Nessuna modifica sostanziale, solo assicurarsi che sia documentato come "Livello 1" nel nuovo schema.

### Livello 2 — Analisi Strutturale del Task

Prima ancora di guardare il Pattern Index, la shield analizza la STRUTTURA di quello che l'utente sta per fare.

Procedura:
1. Dall'input dell'utente, identifica quali sistemi/layer il task tocca. Esempi di layer: frontend, backend/Cloud Functions, Firestore rules, hosting config, auth, external APIs, CI/CD.
2. Conta i layer coinvolti. Se il task tocca più di 2 layer, il rischio strutturale sale.
3. Per ogni interfaccia tra layer (es. frontend ↔ Cloud Functions, Cloud Functions ↔ Firestore), verifica se esiste una skill che copre quel punto di contatto.
4. Interfacce senza skill = gap strutturali predittivi.

Output da aggiungere al report della shield:

```
📐 Structural Analysis
Layers involved: [N] — [layer1], [layer2], [layer3]
Cross-layer interfaces: [N]
Uncovered interfaces: [list of interfaces without a skill]
Structural risk: LOW / MEDIUM / HIGH
```

Criteri:
- 1 layer = LOW
- 2 layer = MEDIUM
- 3+ layer = HIGH
- Interfacce scoperte aggiungono un livello (MEDIUM → HIGH)

Questo livello funziona dal giorno 1, senza storia. Non dipende dal Pattern Index.

### Livello 3 — Consapevolezza Temporale

La shield analizza le CONDIZIONI AMBIENTALI della sessione corrente per stimare il rischio contestuale.

Dati da raccogliere (dal filesystem e dalle quick notes):
- `days_since_last_retrospective_full`: quanti giorni dall'ultimo file `SKILL_AUDIT_*.md`
- `sessions_since_last_audit`: quanti file quick note esistono dopo l'ultimo audit completo
- `days_since_last_deploy`: se ci sono log di deploy recenti nel progetto (opzionale, best-effort)
- `day_of_week`: lunedì dopo pausa vs metà settimana
- `session_gap`: giorni tra l'ultima quick note e oggi (indica pausa dal progetto)

Correlazioni da segnalare:
- `sessions_since_last_audit > 10` → "Sono passate [N] sessioni dall'ultimo audit completo. Storicamente il tasso di errore sale dopo 10+ sessioni senza audit. Consiglio: /retrospective prima di procedere."
- `session_gap > 7` → "Sono passati [N] giorni dall'ultima sessione su questo progetto. Il rischio di errori da contesto perso è elevato. Consiglio: rileggere lo stato del progetto prima di procedere."
- `days_since_last_retrospective_full > 14` → "L'ultimo audit completo risale a [N] giorni fa. Il Pattern Index potrebbe non essere aggiornato."

Output da aggiungere al report:

```
🕐 Temporal Context
Last full audit: [N] days ago
Sessions since audit: [N]
Project gap: [N] days since last session
Temporal risk: LOW / MEDIUM / HIGH
Recommendation: [one line, or "none"]
```

### Livello 4 — Predizione Generativa (Playbook Index)

Non solo "attento qui" ma "ecco come farlo bene". Richiede un nuovo file: `.claude/reports/PLAYBOOK_INDEX.md`.

#### Playbook Index — formato

Il Playbook Index traccia i PERCORSI DI SUCCESSO, non solo gli errori. Ogni entry è un task type con il workflow che ha funzionato.

```markdown
# Playbook Index
<!-- Auto-generated by retrospective. Do not edit manually. -->
<!-- Last updated: [YYYY-MM-DD] -->

## Playbooks

### [playbook-id]: [short description]
- task_type: [keywords that identify this type of task]
- workflow_used: [ordered list of skills invoked]
- steps_taken: [key actions beyond the standard workflow]
- outcome: SUCCESS / PARTIAL / FAILED
- duration_minutes: [estimated time]
- occurrences: [N times this workflow was used for this task type]
- success_rate: [N successes / N total]
- notes: [any useful context]

### firebase-function-deploy: Deploying Firebase Cloud Functions
- task_type: deploy, firebase, cloud-functions, functions
- workflow_used: systematic-debugging → pre-check CORS → test-driven-development → verification-before-completion
- steps_taken: verify CORS middleware before deploy; test with emulator first; deploy single function before batch
- outcome: SUCCESS
- duration_minutes: 15
- occurrences: 8
- success_rate: 7/8
- notes: the one failure was when CORS check was skipped
```

#### Come la shield usa il Playbook Index

Quando il Livello 1 o 2 identifica il tipo di task, il Livello 4 cerca nel Playbook Index un playbook con task_type matching.

Se trova un playbook con success_rate > 75%:

```
✅ Proven Playbook Available
"Firebase function deploy" — 7/8 success rate
Recommended workflow:
1. Verify CORS middleware is configured
2. Test with Firebase emulator
3. Deploy single function first
4. Deploy remaining functions
5. Run verification-before-completion

This workflow succeeded 87% of the time for this task type.
Use it? (The standard brainstorm → plan → execute workflow is always available as fallback)
```

Se trova un playbook con success_rate < 50%:

```
⚠️ Known Difficult Task
"Database migration" — 2/5 success rate
Previous attempts failed when: [extracted from notes]
Consider: [suggestion based on failure analysis]
```

#### Retrospective integration per il Playbook Index

La retrospective deve catturare i successi oltre agli errori. Nella quick mode, aggiungere una sezione:

```markdown
## Successful workflows
- [task description] — workflow: [skills used in order] — duration: ~[N]min — notes: [what made it work]
```

Nella full mode (Phase 1), dopo aver estratto i pattern di errore, estrarre anche i pattern di successo e aggiornare il Playbook Index.

### Livello 5 — Workflow Adattivo

Basandosi sui dati dei livelli 1-4, la shield propone un workflow PERSONALIZZATO per il task corrente, invece di applicare sempre il workflow standard brainstorm → plan → execute.

Logica:
1. Se esiste un playbook con success_rate > 75% per il task type → proponi quel workflow
2. Se il task è semplice (1 layer, nessun pattern di rischio) → proponi workflow abbreviato (skip brainstorming, vai diretto a implement + verify)
3. Se il task è complesso (3+ layer, pattern HIGH risk, o success_rate < 50%) → proponi workflow esteso (brainstorm + plan + execute con checkpoint extra + verify)
4. Se è la prima volta per questo task type (nessun playbook) → usa il workflow standard

Output:

```
🔄 Adaptive Workflow
Based on: [playbook match / structural analysis / default]
Suggested workflow: [skill1] → [skill2] → [skill3]
Rationale: [one sentence explaining why this workflow, not the default]

Differences from standard:
- [what's added/removed/changed and why]
```

L'utente può sempre dire "usa il workflow standard" e la shield non insiste.

### Livello 6 — Community Patterns

La shield, oltre al Pattern Index locale, consulta i pattern della community scaricati dal Cloudflare Worker.

Alla fine del report, se ci sono pattern community rilevanti che NON sono nel Pattern Index locale:

```
🌐 Community Insights — [N] patterns from [M] developers on this stack
• [error description] — reported by [N] developers
  → Prevention: [resolution]
```

I pattern community hanno un campo `reporters` (numero di utenti distinti) che determina la confidence. Solo pattern con `reporters >= 3` vengono mostrati.

I pattern community NON vengono aggiunti automaticamente al Pattern Index locale. Sono advisory. Se l'utente effettivamente incontra l'errore, la retrospective lo aggiunge al Pattern Index locale con un tag `confirmed_from_community: true`.

### Formato del report completo della shield

Tutti i livelli producono un unico report, compatto, presentato PRIMA che qualsiasi lavoro inizi:

```
⚡ Predictive Shield Report

📐 Structure: [N] layers, [risk level]
🕐 Context: [N] days since audit, [risk level]
⚠️ Historical: [N] pattern matches
✅ Playbook: [available/none] — [success rate if available]
🔄 Workflow: [suggested workflow]
🌐 Community: [N] relevant insights

[Details only for HIGH risk items and playbook suggestion]

Pre-activating: [skill1], [skill2]
```

Se TUTTO è LOW risk e non ci sono playbook, il report è una riga:

```
⚡ Predictive Shield — no significant risks detected. Proceeding with standard workflow.
```

Non spammare warning quando non servono.

## 2. BOOTSTRAP AUTOMATICO DEL PATTERN INDEX

Aggiungere a `using-supercycle/SKILL.md` la logica di bootstrap.

### Procedura

1. Al session start, dopo aver caricato using-supercycle, verifica se `.claude/reports/PATTERN_INDEX.md` esiste e contiene almeno 5 pattern.
2. Se il file non esiste o ha meno di 5 pattern → attiva il bootstrap.
3. Il bootstrap rileva lo stack del progetto leggendo:
   - `package.json` → Node.js, dipendenze (firebase, react, express, etc.)
   - `firebase.json` → Firebase project
   - `requirements.txt` / `pyproject.toml` → Python, dipendenze
   - `Cargo.toml` → Rust
   - file di configurazione comuni (`.env`, `docker-compose.yml`, etc.)
   - struttura delle directory (es. `functions/` = Cloud Functions)
4. Da queste informazioni, genera un elenco di stack tag: `["firebase", "vanilla-js", "cloud-functions", "firestore"]`
5. Genera 20-30 pattern nel formato Pattern Index, basati sulla conoscenza di Claude dei problemi comuni per quello stack specifico.
6. Ogni pattern generato ha un campo aggiuntivo: `source: bootstrap` (per distinguerlo dai pattern derivati dall'esperienza reale dell'utente che avranno `source: session`).
7. Scrivi il Pattern Index.
8. Mostra all'utente: "Ho analizzato il tuo stack ([stack tags]) e generato [N] pattern predittivi iniziali. La Predictive Shield è attiva. I pattern verranno affinati dalla tua esperienza reale nelle prossime sessioni."

### Pattern bootstrap vs pattern reali

I pattern `source: bootstrap` hanno regole diverse:
- Se non vengono mai confermati da un errore reale dopo 90 giorni → marcati `unconfirmed` e la shield smette di usarli
- Se un errore reale corrisponde a un pattern bootstrap → il pattern cambia `source: confirmed` e il suo peso aumenta
- I pattern `source: session` (dall'esperienza reale) hanno sempre priorità nel report della shield rispetto ai `source: bootstrap`

## 4. CLOUDFLARE WORKER

Creare il worker in una directory `infrastructure/cloudflare-worker/`.

### Struttura

```
infrastructure/
  cloudflare-worker/
    wrangler.toml          # config Cloudflare
    src/
      index.js             # il worker (max 100 righe)
    README.md              # istruzioni di deploy per il maintainer
```

### Endpoint POST `/patterns`

Riceve:
```json
{
  "stack": "firebase",
  "user_hash": "a1b2c3...",
  "patterns": [
    {
      "id": "cors-firebase-functions",
      "triggers": ["deploy", "firebase", "cloud-functions", "cors"],
      "error": "CORS header missing after function deployment",
      "resolution": "Add CORS middleware before handler",
      "severity": "HIGH",
      "occurrences": 5
    }
  ],
  "supercycle_version": "1.0.0"
}
```

Validazione:
- `stack` deve essere una stringa alfanumerica, max 50 char
- `user_hash` deve essere una stringa hex, esattamente 64 char (SHA-256)
- `patterns` deve essere un array, max 50 elementi
- Ogni pattern deve avere `id`, `triggers`, `error`, `resolution`, `severity`
- Nessun campo può contenere path di file (regex: niente `/` o `\` in error/resolution)
- Nessun campo può contenere nomi che sembrano path (regex: niente `.js`, `.py`, `.ts`, `.md` etc. in error/resolution)
- Se la validazione fallisce → 400 con messaggio chiaro
- Rate limit: max 10 POST per user_hash per ora

Azione dopo validazione:
- Committa il payload come file JSON in `community-patterns/inbox/[stack]/[timestamp]-[user_hash_prefix].json` nel repo `albertocontardi/supercycle` via GitHub API
- Usa il token GitHub del maintainer (nei secrets del worker: `GITHUB_TOKEN`)
- Risponde 201 con `{ "status": "accepted", "patterns_received": N }`

### Endpoint GET `/patterns/:stack`

Parametri:
- `:stack` — tag dello stack (es. `firebase`, `react`, `python`)
- Query param `min_reporters` — minimo numero di utenti che hanno riportato il pattern (default: 3)

Azione:
- Fetch il file `community-patterns/aggregated/[stack].json` dal repo via GitHub raw content
- Cache nella edge Cloudflare con TTL 1 ora
- Se il file non esiste → 404 con `{ "status": "no_patterns", "stack": "[stack]" }`

Risponde:
```json
{
  "stack": "firebase",
  "updated_at": "2026-03-17T10:00:00Z",
  "pattern_count": 25,
  "patterns": [
    {
      "id": "cors-firebase-functions",
      "triggers": ["deploy", "firebase", "cloud-functions", "cors"],
      "error": "CORS header missing after function deployment",
      "resolution": "Add CORS middleware before handler",
      "severity": "HIGH",
      "reporters": 15,
      "first_seen": "2026-01-10",
      "last_seen": "2026-03-15"
    }
  ]
}
```

### Sicurezza

- Il worker NON richiede autenticazione per GET (dati pubblici)
- Il worker NON richiede autenticazione per POST (i pattern sono anonimi, il rate limit previene abuse)
- Il GITHUB_TOKEN ha SOLO permesso `contents: write` sul repo `albertocontardi/supercycle`
- Il worker rifiuta payload > 50KB
- Il worker logga solo contatori (N richieste, N errori), mai il contenuto dei pattern

### wrangler.toml

```toml
name = "supercycle-patterns"
main = "src/index.js"
compatibility_date = "2024-01-01"

[vars]
GITHUB_REPO = "albertocontardi/supercycle"
GITHUB_BRANCH = "main"

# GITHUB_TOKEN va nei secrets: wrangler secret put GITHUB_TOKEN
```

## 5. GITHUB ACTION DI AGGREGAZIONE

Creare `.github/workflows/aggregate-patterns.yml` nel repo supercycle.

### Trigger

Si attiva quando un file viene committato in `community-patterns/inbox/`.

### Logica

1. Leggi tutti i file JSON in `community-patterns/inbox/[stack]/`
2. Per ogni stack presente:
   a. Leggi il file aggregato esistente `community-patterns/aggregated/[stack].json` (o creane uno vuoto)
   b. Per ogni pattern nel file inbox:
      - Se il pattern ID esiste già nell'aggregato: incrementa `reporters` (deduplicando per user_hash), aggiorna `last_seen`
      - Se il pattern ID è nuovo: aggiungilo con `reporters: 1`
   c. Scrivi il file aggregato aggiornato
3. Svuota `community-patterns/inbox/` (elimina i file processati)
4. Committa e pusha i file aggregati aggiornati

### Deduplicazione

Un singolo user_hash conta come 1 reporter per pattern, anche se invia lo stesso pattern 100 volte. Lo user_hash va tracciato in un campo interno `_reporter_hashes` nel file aggregato (non esposto nella GET del worker).

### Struttura directory

```
community-patterns/
  inbox/                           # file temporanei dal worker, svuotati dopo aggregazione
    firebase/
      2026-03-17T10-00-00-a1b2c3.json
    react/
      ...
  aggregated/                      # file finali serviti dal worker
    firebase.json
    react.json
    python.json
```

## 6. OPT-IN FLOW

### Prima installazione

Quando `using-supercycle` si attiva per la prima volta E il file `.claude/supercycle-config.json` non esiste:

1. Dopo il bootstrap del Pattern Index, mostra:

```
SuperCycle can share anonymous error patterns with the community to improve predictions for everyone.

What gets shared:
- Error patterns: keywords, error description, resolution, severity
- Stack tags: e.g. "firebase", "react"
- An anonymous hash (no username, no email, no project name)

What NEVER gets shared:
- Source code
- File paths
- Project names
- Personal information
- Any content from your codebase

This data helps other developers avoid errors you've already solved, and you benefit from errors they've solved.

Enable community pattern sharing?
```

2. Salva la scelta in `.claude/supercycle-config.json`:

```json
{
  "community_sharing": true,
  "opted_in_at": "2026-03-17T10:00:00Z",
  "supercycle_version": "1.0.0",
  "user_hash": "a1b2c3..."
}
```

Il `user_hash` viene generato una volta sola: `SHA-256(hostname + username + "supercycle-salt")`. Non cambia mai. Non è reversibile.

3. Se l'utente dice no:

```json
{
  "community_sharing": false,
  "opted_out_at": "2026-03-17T10:00:00Z"
}
```

Tutto funziona identicamente, ma la retrospective non invia pattern al worker e la shield non consulta i pattern community.

### Cambio preferenza

L'utente può sempre cambiare editando `.claude/supercycle-config.json` o dicendo "disable community sharing" / "enable community sharing" in sessione.

### Quando avviene la condivisione

La condivisione avviene durante la retrospective FULL mode (Phase 1), dopo aver aggiornato il Pattern Index locale. NON durante la quick mode — la quick mode è veloce e non deve fare network call.

Nella full mode, se `community_sharing: true`:
1. Raccogli tutti i pattern dal Pattern Index locale che hanno `source: session` o `source: confirmed` (NON i `source: bootstrap` — quelli sono generati da Claude, non dall'esperienza dell'utente)
2. Anonimizza: rimuovi qualsiasi campo che possa contenere info di progetto
3. POST al worker endpoint con stack tag e user_hash
4. Logga nel report: "Shared [N] patterns with community"

Nella full mode, sempre (indipendente dall'opt-in):
1. GET dal worker endpoint con lo stack tag del progetto
2. Salva i pattern community in `.claude/reports/COMMUNITY_PATTERNS.md` (cache locale)
3. La shield li usa al prossimo session start

**Nota: la ricezione dei community patterns avviene SEMPRE**, anche se l'utente ha fatto opt-out dalla condivisione. L'opt-out riguarda solo l'invio. Ricevere è sempre attivo perché non invia dati dell'utente.
